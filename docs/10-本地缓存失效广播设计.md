# Phase 9：本地缓存失效广播设计

## 1. 阶段目标

Phase 9 的目标是补齐多节点本地缓存一致性链路。

当前项目已经具备：

```text
访问上报
  -> rsyslog
  -> Kafka
  -> tmc-server 热点探测
  -> HotKeySnapshot 写入 etcd
  -> SDK 监听热点快照
  -> HotKeyManager 更新热点集合
  -> 热点 key 读请求走本地缓存
```

但当业务应用有多个实例时，还存在一个关键问题：

```text
实例 A 写 Redis 后，只删除了实例 A 的本地缓存；
实例 B、实例 C 可能仍然持有同一个 key 的旧本地缓存。
```

本阶段要实现：

```text
TmcJedis 写操作
  -> Redis 写成功
  -> 当前节点删除本地缓存
  -> 发布 InvalidationEvent 到 etcd
  -> 其他 SDK 节点监听失效事件
  -> 删除自己的 Caffeine 本地缓存
```

完成后应具备以下能力：

1. `TmcJedis.set/del/expire` 等写操作成功后可以发布本地缓存失效事件。
2. 当前节点写成功后会先删除自己的本地缓存。
3. 失效事件写入 etcd 的唯一事件路径，避免多次写操作互相覆盖。
4. SDK 可以监听当前应用的失效事件前缀。
5. SDK 收到其他节点发布的失效事件后删除本地缓存。
6. SDK 收到自己发布的失效事件时跳过处理，避免重复删除。
7. 非法事件、etcd 异常、监听中断不能影响业务 Redis 读写主流程。
8. 单元测试覆盖发布、监听、自身事件跳过、其他节点事件生效和自动配置。

## 2. 原文思路对齐

有赞原文中，本地缓存不仅要能因为热点 key 下发而生效，还要能在数据变更时被及时失效。

本项目使用 etcd 复现本地缓存失效广播：

| 原文能力 | 本项目实现 |
|---|---|
| 业务写操作触发本地缓存失效 | `TmcJedis` 写 Redis 成功后调用 `TmcClient.invalidate(...)` |
| 失效事件广播给其他客户端 | `InvalidationReporter` 写入 etcd |
| SDK 监听失效事件 | `InvalidationListener` watch etcd 前缀 |
| 其他节点删除本地缓存 | `InvalidationListener` 调用 `CaffeineLocalCache.invalidate(key)` |
| 避免本节点重复处理 | 通过 `clientId` 过滤自己发布的事件 |

本阶段完成后，TMC 的核心闭环变为：

```text
读热点 key
  -> 本地缓存减轻 Redis 压力

写热点 key
  -> 失效广播清理各节点本地缓存
```

## 3. 阶段边界

本阶段实现：

- SDK 失效事件发布。
- SDK 失效事件监听。
- `TmcJedis` 写操作接入失效事件发布。
- 当前节点本地缓存删除。
- 其他节点本地缓存删除。
- 自身事件过滤。
- etcd 事件 TTL。
- 通信线程隔离。
- 指标和单元测试。

本阶段不实现：

- 服务端参与失效事件转发。
- 批量 key 或通配符失效。
- Redis Lua 写操作封装。
- 失效事件持久化审计。
- 管理后台手工失效入口。

## 4. etcd Key 设计

失效事件使用已有 `EtcdKeys.invalidationEventPath(appName, eventId)` 生成路径。

```text
/tmc/invalidation-events/{appName}/{eventId}
```

监听前缀使用 `EtcdKeys.invalidationEventPrefix(appName)`。

```text
/tmc/invalidation-events/{appName}/
```

示例：

```text
/tmc/invalidation-events/product-service/7f6f3d7a-2c8a-4dd2-b95e-1d889c5e933e
```

设计原则：

- 失效事件是“事件”，不是“状态”。
- 每次写操作都必须生成新的 `eventId`。
- 不能像热点快照一样覆盖同一个 key。
- 事件写入 etcd 时绑定短 TTL，避免历史事件无限堆积。
- SDK 使用前缀 watch，确保同一个 app 下的每个事件都能被看到。

热点快照与失效事件的区别：

| 类型 | etcd 路径 | 是否覆盖 | 原因 |
|---|---|---|---|
| 热点快照 | `/tmc/hotkeys/{appName}` | 覆盖 | 表达当前最新热点状态 |
| 失效事件 | `/tmc/invalidation-events/{appName}/{eventId}` | 不覆盖 | 表达每次独立写操作 |

## 5. 数据协议

etcd value 使用 `InvalidationEvent` JSON。

示例：

```json
{
  "appName": "product-service",
  "key": "product:10001",
  "operation": "SET",
  "clientId": "tmc-client-a",
  "timestamp": 1720000003000,
  "eventId": "7f6f3d7a-2c8a-4dd2-b95e-1d889c5e933e"
}
```

字段说明：

| 字段 | 说明 |
|---|---|
| `appName` | 业务应用名称，只允许同 app 的 SDK 处理 |
| `key` | 需要删除本地缓存的 Redis key |
| `operation` | 导致失效的写操作类型，例如 `SET`、`DEL`、`EXPIRE` |
| `clientId` | 发布事件的 SDK 节点 ID，用于自身事件过滤 |
| `timestamp` | 事件发生时间，单位毫秒 |
| `eventId` | 事件唯一 ID，用于生成 etcd 唯一路径 |

校验规则：

| 字段 | 规则 |
|---|---|
| `appName` | 非空，且必须等于当前 SDK 配置的 `tmc.app-name` |
| `key` | 非空 |
| `operation` | 非空 |
| `clientId` | 非空 |
| `timestamp` | 非空且大于 0 |
| `eventId` | 非空 |

## 6. 模块结构

本阶段主要涉及 `tmc-sdk` 和 `tmc-jedis`。

### 6.1 `tmc-sdk`

建议新增或修改：

```text
cn.ayice.tmc.communication.InvalidationProperties
cn.ayice.tmc.communication.InvalidationReporter
cn.ayice.tmc.communication.InvalidationListener
cn.ayice.tmc.hotkey.CaffeineLocalCache
cn.ayice.tmc.sdk.TmcClient
cn.ayice.tmc.sdk.TmcAutoConfiguration
cn.ayice.tmc.sdk.TmcMetrics
cn.ayice.tmc.sdk.TmcProperties
```

职责说明：

| 类 | 职责 |
|---|---|
| `InvalidationProperties` | 保存失效广播开关、事件 TTL、重连间隔等配置 |
| `InvalidationReporter` | 写操作成功后把 `InvalidationEvent` 写入 etcd |
| `InvalidationListener` | watch 当前 app 的失效事件前缀并删除本地缓存 |
| `CaffeineLocalCache` | 提供单 key 删除能力 |
| `TmcClient` | 提供写路径失效入口，组合本地删除和事件发布 |
| `TmcAutoConfiguration` | 自动创建失效发布器和监听器 |
| `TmcMetrics` | 只保留核心效果指标，失效发布和监听不进入 Grafana 主指标 |
| `TmcProperties` | 挂载失效广播配置 |

说明：

- 不新增 Java `interface`。
- `InvalidationReporter` 和 `InvalidationListener` 是 concrete class。
- etcd 仍然属于 SDK 通信模块。
- 本地缓存删除仍然由热点模块的 `CaffeineLocalCache` 执行。

### 6.2 `tmc-jedis`

建议修改：

```text
cn.ayice.tmc.jedis.TmcJedis
```

职责说明：

| 类 | 职责 |
|---|---|
| `TmcJedis` | 写 Redis 成功后调用 `TmcClient.invalidateAfterWrite(key, operation)` |

`tmc-jedis` 不直接依赖 etcd，不直接构造 `InvalidationEvent`。

原因：

- `tmc-jedis` 只负责 Jedis 风格适配。
- 失效广播属于 SDK 通信能力。
- 这样后续如果支持更多 Redis 写方法，仍然复用同一个 SDK 失效入口。

## 7. 配置设计

### 7.1 SDK 配置示例

```yaml
tmc:
  app-name: product-service
  client-id: tmc-client-a
  etcd:
    endpoints:
      - http://localhost:2379
    connect-timeout-millis: 3000
  invalidation:
    enabled: true
    report-enabled: true
    listen-enabled: true
    event-ttl-seconds: 30
    reconnect-delay-millis: 3000
```

### 7.2 字段说明

| 字段 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `enabled` | `boolean` | `true` | 是否启用失效广播整体能力 |
| `reportEnabled` | `boolean` | `true` | 是否发布本节点写操作失效事件 |
| `listenEnabled` | `boolean` | `true` | 是否监听其他节点失效事件 |
| `eventTtlSeconds` | `long` | `30` | 失效事件在 etcd 中保留多久 |
| `reconnectDelayMillis` | `long` | `3000` | watch 异常后的重连等待时间 |

约束：

- `eventTtlSeconds > 0`。
- `reconnectDelayMillis > 0`。
- `eventTtlSeconds` 不需要很长，只要覆盖正常 watch 延迟即可。

### 7.3 开关关系

```text
tmc.enabled=false
  -> 不创建任何 TMC 组件

tmc.invalidation.enabled=false
  -> 不发布也不监听失效事件

tmc.invalidation.report-enabled=false
  -> 写操作只删除当前节点本地缓存，不广播其他节点

tmc.invalidation.listen-enabled=false
  -> 当前节点不处理其他节点广播
```

## 8. 写路径设计

### 8.1 当前写路径

当前 `TmcJedis` 写路径大致是：

```text
set(key, value)
  -> jedis.set(key, value)
  -> result == OK
  -> tmcClient.invalidate(key)
```

`del` 和 `expire` 类似。

### 8.2 本阶段写路径

本阶段建议调整为：

```text
set(key, value)
  -> jedis.set(key, value)
  -> result == OK
  -> tmcClient.invalidateAfterWrite(key, CacheOperation.SET)
```

`invalidateAfterWrite` 内部流程：

```text
invalidateAfterWrite(key, operation)
  -> localCache.invalidate(key)
  -> invalidationReporter.report(key, operation)
```

重要约束：

- 只有 Redis 写成功后才发布失效事件。
- Redis 写失败或返回未生效结果时不发布失效事件。
- 本地删除应该先执行，保证当前节点后续读不会命中旧值。
- 发布失效事件失败不能影响 Redis 写操作返回结果。

### 8.3 写操作成功判断

| 方法 | Redis 成功条件 | 失效 operation |
|---|---|---|
| `set` | 返回 `"OK"` | `CacheOperation.SET` |
| `del` | 返回值 `> 0` | `CacheOperation.DEL` |
| `expire` | 返回值 `== 1` | `CacheOperation.EXPIRE` |

## 9. `InvalidationReporter` 设计

职责：

- 根据 `appName`、`clientId`、`key`、`operation` 构造 `InvalidationEvent`。
- 生成全局唯一 `eventId`。
- 使用 `JsonUtils.toJson(event)` 序列化。
- 写入 `EtcdKeys.invalidationEventPath(appName, eventId)`。
- 写入时绑定 lease TTL。
- 捕获异常并记录指标，不能影响写操作主流程。

建议方法：

```java
public class InvalidationReporter implements AutoCloseable {
    public void report(String key, CacheOperation operation) {
    }

    @Override
    public void close() {
    }
}
```

发布流程：

```text
report(key, operation)
  -> 校验 key 和 operation
  -> eventId = UUID.randomUUID().toString()
  -> event = new InvalidationEvent(appName, key, operation, clientId, now, eventId)
  -> path = EtcdKeys.invalidationEventPath(appName, eventId)
  -> value = JsonUtils.toJson(event)
  -> lease.grant(eventTtlSeconds)
  -> kv.put(path, value, leaseId)

```

异常处理：

```text
etcd 写入失败

  -> 记录 warn 日志
  -> 返回，不抛出
```

## 10. `InvalidationListener` 设计

职责：

- 监听 `EtcdKeys.invalidationEventPrefix(appName)`。
- 收到 PUT 事件后解析 `InvalidationEvent`。
- 校验事件字段。
- 忽略其他 app 的事件。
- 忽略自己 `clientId` 发布的事件。
- 对其他节点事件调用 `CaffeineLocalCache.invalidate(key)`。
- 监听异常后重连。
- 关闭时释放 watch 和线程池。

建议方法：

```java
public class InvalidationListener implements SmartLifecycle, AutoCloseable {
    public void start() {
    }

    public void stop() {
    }

    @Override
    public void close() {
    }
}
```

事件处理流程：

```text
watch PUT event
  -> value bytes 转字符串
  -> JsonUtils.fromJson(value, InvalidationEvent.class)
  -> validate event
  -> event.appName != localAppName，忽略
  -> event.clientId == localClientId，忽略
  -> localCache.invalidate(event.key)

```

DELETE 事件处理：

```text
watch DELETE event
  -> 忽略
```

原因：

- 失效事件 key 绑定 TTL 后会自动删除。
- TTL 删除只代表历史事件过期，不代表需要再次删除本地缓存。

## 11. 自身事件过滤

为什么要过滤自身事件：

```text
当前节点写 Redis 成功
  -> 已经删除当前节点本地缓存
  -> 又收到自己写入 etcd 的事件
  -> 再删一次没有意义
```

过滤规则：

```text
event.clientId == tmc.client-id

  -> return
```

注意：

- `clientId` 必须稳定标识一个 SDK 实例。
- 当前 `TmcProperties.clientId` 默认使用 UUID，可以区分不同进程。
- 如果业务方显式配置相同 `clientId`，会导致实例之间互相误判为自身事件，所以文档和配置说明中应提示避免重复。

## 12. 线程隔离设计

失效广播属于通信模块，不能占用业务读写线程。

建议：

```text
InvalidationReporter
  -> 写操作线程调用 report
  -> report 内部只做快速入队
  -> 后台线程批量或逐条写 etcd

InvalidationListener
  -> 独立 watch 线程
  -> 收到事件后快速删除本地缓存
```

本阶段为了实现清晰，可以先让 `InvalidationReporter.report(...)` 直接写 etcd，但必须满足：

- 捕获异常。
- 不影响 Redis 写操作返回。
- 后续如果写入延迟明显，再改为有界队列 + 后台线程。

从完整设计角度，推荐最终实现为：

```text
业务写线程
  -> 当前节点 localCache.invalidate(key)
  -> InvalidationReporter.offer(event)
  -> 立即返回

后台通信线程
  -> 从有界队列取事件
  -> 写入 etcd
```

队列满时：

```text
丢弃失效事件

  -> 打 warn 日志
```

说明：

- 丢失失效事件会影响其他节点本地缓存一致性。
- 但不能为了广播阻塞业务写主流程。
- 后续可通过较短本地缓存 TTL 和监控告警降低风险。

## 13. 自动配置设计

`TmcAutoConfiguration` 新增：

```text
InvalidationReporter
InvalidationListener
```

创建条件：

```text
tmc.enabled=true
tmc.invalidation.enabled=true
tmc.invalidation.report-enabled=true
不存在用户自定义 InvalidationReporter
```

```text
tmc.enabled=true
tmc.invalidation.enabled=true
tmc.invalidation.listen-enabled=true
存在 CaffeineLocalCache
不存在用户自定义 InvalidationListener
```

建议：

```java
@Bean
@ConditionalOnMissingBean
@ConditionalOnProperty(
        prefix = "tmc.invalidation",
        name = {"enabled", "report-enabled"},
        havingValue = "true",
        matchIfMissing = true
)
public InvalidationReporter invalidationReporter(...) {
}
```

```java
@Bean
@ConditionalOnMissingBean
@ConditionalOnBean(CaffeineLocalCache.class)
@ConditionalOnProperty(
        prefix = "tmc.invalidation",
        name = {"enabled", "listen-enabled"},
        havingValue = "true",
        matchIfMissing = true
)
public InvalidationListener invalidationListener(...) {
}
```

如果 Spring 条件注解不适合同时表达多个属性，可以拆成更直接的条件或在配置类中显式判断。

## 14. 指标设计

`TmcMetrics` 建议新增：

| 指标 | 含义 |
|---|---|
| `localInvalidations` | 当前节点主动删除本地缓存次数 |
| `invalidationReportQueued` | 失效事件成功进入发布队列次数 |
| `invalidationReportDropped` | 发布队列满导致丢弃次数 |
| `invalidationReportSucceeded` | 失效事件写入 etcd 成功次数 |
| `invalidationReportFailed` | 失效事件写入 etcd 失败次数 |
| `invalidationReceived` | 收到并处理其他节点失效事件次数 |
| `invalidationSelfIgnored` | 忽略自身失效事件次数 |
| `invalidationInvalid` | 非法失效事件次数 |
| `invalidationWatchReconnect` | watch 异常后重连次数 |
| `invalidationWatchFailed` | watch 启动或处理失败次数 |

这些指标用于判断：

- 写操作后是否产生失效事件。
- 其他节点是否收到失效事件。
- 是否存在大量非法事件。
- etcd watch 是否频繁断开。
- 发布队列是否容量不足。

## 15. 测试设计

### 15.1 `InvalidationReporterTest`

覆盖：

1. `report(key, operation)` 会写入 `/tmc/invalidation-events/{appName}/{eventId}`。
2. 写入 value 是合法 `InvalidationEvent` JSON。
3. 每次 report 都生成不同 `eventId`。
4. 写入时绑定 lease TTL。
5. key 为空时不写入并记录失败或非法指标。
6. operation 为空时不写入并记录失败或非法指标。
7. etcd 写入异常不会向外抛出。

### 15.2 `InvalidationListenerTest`

覆盖：

1. 收到其他节点 PUT 事件会删除本地缓存。
2. 收到自身 `clientId` 事件会跳过。
3. 收到其他 app 事件会跳过。
4. 收到非法 JSON 会记录 invalid 指标。
5. 收到字段缺失事件会记录 invalid 指标。
6. 收到 DELETE 事件会忽略。
7. watch 异常后会记录重连指标。
8. `close()` 会释放 watch、client 和线程池。

### 15.3 `TmcClientTest`

补充：

1. `invalidateAfterWrite(key, SET)` 会删除本地缓存。
2. 有 `InvalidationReporter` 时会发布失效事件。
3. 无 `InvalidationReporter` 时只删除本地缓存。
4. `InvalidationReporter` 抛异常时不影响方法返回。

### 15.4 `TmcJedisTest`

补充：

1. `set` 返回 `"OK"` 时调用 `invalidateAfterWrite(key, SET)`。
2. `set` 未成功时不触发失效。
3. `del` 返回 `> 0` 时调用 `invalidateAfterWrite(key, DEL)`。
4. `del` 返回 `0` 时不触发失效。
5. `expire` 返回 `1` 时调用 `invalidateAfterWrite(key, EXPIRE)`。
6. `expire` 返回 `0` 时不触发失效。

### 15.5 `TmcAutoConfigurationTest`

补充：

1. 默认配置下创建 `InvalidationReporter`。
2. 默认配置下创建 `InvalidationListener`。
3. `tmc.invalidation.enabled=false` 时两者都不创建。
4. `tmc.invalidation.report-enabled=false` 时不创建 reporter。
5. `tmc.invalidation.listen-enabled=false` 时不创建 listener。
6. 用户自定义同类型 Bean 时不覆盖。

## 16. 开发顺序

建议按以下顺序开发：

1. 在 `tmc-sdk` 增加 `InvalidationProperties`。
2. 修改 `TmcProperties`，挂载 `invalidation` 配置。
3. 修改失效广播测试，验证行为本身，不新增失效广播指标断言。
4. `TmcMetrics` 和 `TmcMetricsSnapshot` 继续只保留核心效果指标。
5. 编写 `InvalidationReporterTest`。
6. 实现 `InvalidationReporter`。
7. 编写 `InvalidationListenerTest`。
8. 实现 `InvalidationListener`。
9. 修改 `TmcClientTest`，补充 `invalidateAfterWrite` 行为。
10. 修改 `TmcClient`，新增 `invalidateAfterWrite(key, operation)`。
11. 修改 `TmcJedisTest`，补充写操作 operation 断言。
12. 修改 `TmcJedis`，写成功后调用 `invalidateAfterWrite`。

## 18. 联调验证

本阶段完成后，可以用两个 demo 实例验证多节点失效：

```text
demo-a clientId=client-a
demo-b clientId=client-b
appName 相同
etcd endpoints 相同
```

验证流程：

1. 启动 Redis 和 etcd。
2. 启动 demo-a 和 demo-b。
3. 让目标 key 进入热点集合，并在两个实例中读取一次，形成本地缓存。
4. 在 demo-a 执行 `set(key, newValue)`。
5. demo-a 写 Redis 成功后删除自己的本地缓存，并发布 `InvalidationEvent`。
6. demo-b 监听到事件后删除自己的本地缓存。
7. demo-b 再次 `get(key)`，应回源 Redis 读取新值并重新写入本地缓存。

完整链路：

```text
demo-a TmcJedis.set(key, value)
  -> Redis set OK
  -> demo-a localCache.invalidate(key)
  -> InvalidationReporter 写 etcd
  -> demo-b InvalidationListener 收到事件
  -> demo-b localCache.invalidate(key)
  -> demo-b 后续 get 回源 Redis
```

## 19. 风险与处理

| 风险 | 处理方式 |
|---|---|
| etcd 写入失败 | 记录失败指标，不影响 Redis 写返回 |
| watch 断开 | 独立线程重连，并记录重连指标 |
| 失效事件丢失 | 依赖本地缓存 TTL 兜底，并通过日志、测试和手动验证排查 |
| 事件积压 | etcd key 绑定短 TTL |
| 自身事件重复处理 | 使用 `clientId` 过滤 |
| 多实例配置相同 `clientId` | 文档提示避免重复，后续可在启动日志中提示 |
| 非法事件导致监听中断 | 捕获异常、记录 invalid 指标、继续监听 |
| 写操作广播耗时 | 后续可将 reporter 改为有界队列 + 后台线程 |
