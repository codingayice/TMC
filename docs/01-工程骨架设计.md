# 阶段一：TMC 项目工程骨架设计文档

## 一、阶段目标

阶段一的目标是从 0 搭建 TMC 项目的基础工程结构，为后续开发热点 Key 探测、本地缓存、Jedis 透明接入、服务端热度统计、热点推送、失效广播和压测验证提供稳定的工程基础。

本阶段不实现完整业务功能，只完成以下基础能力：

1. 创建 TMC 多模块工程。
2. 明确各模块职责边界。
3. 定义模块之间的依赖关系。
4. 统一基础技术栈和版本管理。
5. 搭建本地 Redis、etcd、服务端、Demo 运行环境。
6. 提供后续阶段可以继续开发的代码骨架。
7. 建立基础编码规范、日志规范、测试规范。

阶段一完成后，项目应该具备以下状态：

```text
项目可以编译
基础模块已经创建
本地依赖环境可以启动
Demo 服务可以访问 Redis
服务端骨架可以启动
后续阶段可以直接在此基础上开发
```

---

## 二、项目背景

TMC 的目标是解决缓存热点 Key 问题。

在高并发场景下，某些 Redis Key 会被大量访问，例如：

```text
商品详情 Key
活动配置 Key
秒杀库存展示 Key
首页推荐 Key
营销规则 Key
```

如果所有请求都直接访问 Redis，热点 Key 可能导致：

```text
Redis 单分片 QPS 过高
缓存集群负载不均
网络带宽被打满
应用响应时间上升
Redis 节点出现慢查询或超时
```

TMC 的核心思路是：

```text
客户端上报 Key 访问事件
服务端统计热点 Key
服务端将热点 Key 推送给客户端
客户端将热点 Key 缓存在本地
后续热点读请求直接命中本地缓存
写操作触发本地缓存失效和集群失效广播
```

因此，完整 TMC 系统会包含以下部分：

```text
公共协议模块
客户端 SDK
Jedis 透明接入模块
热点探测服务端
Demo 应用
压测工具
```

阶段一只负责把这些模块的工程结构先搭建出来。

---

## 三、总体架构

完整系统的目标架构如下：

```text
业务应用
  ↓
tmc-jedis
  ↓
tmc-sdk
  ↓
本地缓存 / 热点 Key 表 / 访问事件上报 / 失效监听
  ↓
Redis

tmc-sdk
  ↓
访问事件上报
  ↓
tmc-server
  ↓
热点统计 / TopN 探测
  ↓
etcd
  ↓
tmc-sdk 接收热点 Key / 失效事件
```

阶段一只搭建工程骨架，不要求完整跑通上述链路。

阶段一需要完成的最小结构：

```text
tmc-parent
├── tmc-common
├── tmc-sdk
├── tmc-jedis
├── tmc-server
├── tmc-demo
└── tmc-benchmark
```

## 四、模块拆分设计

### 1. tmc-parent

#### 模块定位

`tmc-parent` 是整个项目的父工程，负责统一管理：

```text
子模块
依赖版本
插件版本
Java 版本
构建配置
测试配置
代码风格基础规则
```

#### 职责

```text
1. 声明所有子模块
2. 统一 dependencyManagement
3. 统一 pluginManagement
4. 统一项目版本号
5. 统一 Java 编译版本
```

#### 不负责

```text
不写业务代码
不实现 SDK 逻辑
不实现服务端逻辑
不实现缓存逻辑
```

---

### 2. tmc-common

#### 模块定位

`tmc-common` 是公共基础模块，被客户端、服务端和 Jedis 接入模块共同依赖。

#### 职责

```text
1. 公共数据模型
2. 公共枚举
3. 公共常量
4. 公共配置类
5. 公共异常
6. JSON 序列化工具
7. 路径生成工具
```

#### 后续会承载的数据模型

```text
KeyAccessEvent       访问事件
HotKeyEntry          热点 Key 信息
InvalidateEvent      缓存失效事件
TmcConfig            总配置
LocalCacheConfig     本地缓存配置
ReportConfig         上报配置
EtcdConfig           etcd 配置
KeyRuleConfig        Key 黑白名单配置
```

#### 不负责

```text
不做热点统计
不做本地缓存
不做 Jedis 改造
不做 etcd watch
不做 Redis 访问
```

---

### 3. tmc-sdk

#### 模块定位

`tmc-sdk` 是客户端核心模块，负责应用侧的 TMC 能力。

#### 职责

```text
1. 热点 Key 管理
2. 本地缓存管理
3. 访问事件异步上报
4. 热点 Key 推送监听
5. 失效事件监听
6. 本机缓存失效
7. SDK 生命周期管理
8. SDK 降级保护
```

#### 对外提供的核心接口

```java
public interface TmcSdk {

    String get(String key, Callable<String> redisLoader);

    List<String> mget(
            List<String> keys,
            Function<List<String>, List<String>> redisLoader
    );

    void reportAccessAsync(String key);

    void invalidate(String key, InvalidateEventType eventType);

    boolean isEnabled();

    void shutdown();
}
```

#### 不负责

```text
不直接改造 Jedis
不启动服务端
不做全局热点统计
不负责 Redis 连接池管理
```

---

### 4. tmc-jedis

#### 模块定位

`tmc-jedis` 是 Redis 客户端接入层，负责让业务原有 Jedis 访问透明经过 TMC SDK。

#### 职责

```text
1. 改造 JedisPool
2. 改造 JedisFactory
3. 改造 Jedis 的 get / mget 命令
4. 改造 Jedis 的 set / del / expire 命令
5. 将读请求交给 TmcSdk 判断是否命中本地缓存
6. 将写请求交给 TmcSdk 做缓存失效
7. 保证 TMC 异常时回退到原生 Redis 访问
```

#### 阶段一只做

```text
创建模块
定义包结构
预留类
不做源码级改造
```

#### 不负责

```text
不实现热点统计
不实现本地缓存细节
不实现服务端接口
```

---

### 5. tmc-server

#### 模块定位

`tmc-server` 是热点探测服务端。

#### 职责

```text
1. 接收客户端上报的 Key 访问事件
2. 按 appName + key 统计访问热度
3. 维护滑动窗口
4. 计算 TopN 热点 Key
5. 将热点 Key 推送给客户端
6. 暴露监控指标
```

#### 阶段一只做

```text
创建 Spring Boot 工程骨架
提供健康检查接口
提供访问事件接收接口占位
```

#### 不负责

```text
阶段一不实现滑动窗口
阶段一不实现 TopN
阶段一不实现热点推送
```

---

### 6. tmc-demo

#### 模块定位

`tmc-demo` 是演示应用，用于验证 TMC 完整链路。

#### 职责

```text
1. 提供 HTTP 接口访问 Redis
2. 模拟业务 get / set / del 操作
3. 后续接入 tmc-jedis
4. 用于端到端联调
```

#### 阶段一接口

```text
GET    /demo/cache/get?key=item:10001
POST   /demo/cache/set?key=item:10001&value=hello
DELETE /demo/cache/del?key=item:10001
```

#### 不负责

```text
不做热点统计
不做压测
不实现 SDK 内部逻辑
```

---

### 7. tmc-benchmark

#### 模块定位

`tmc-benchmark` 是压测模块，用于后续验证 TMC 效果。

#### 后续职责

```text
1. 模拟单热点 Key 访问
2. 模拟多热点 Key 访问
3. 模拟均匀 Key 访问
4. 验证 Redis QPS 是否下降
5. 验证本地缓存命中率
6. 验证热点识别延迟
7. 验证失效广播延迟
```

#### 阶段一只做

```text
创建模块
编写 README
预留压测入口类
```

## 五、模块依赖关系

模块依赖必须保持单向，避免循环依赖。

推荐依赖关系：

```text
tmc-common
   ↑
   ├── tmc-sdk
   │      ↑
   │      └── tmc-jedis
   │
   ├── tmc-server
   │
   ├── tmc-demo
   │      └── tmc-jedis
   │
   └── tmc-benchmark
```

具体规则：

```text
tmc-common 不依赖任何 TMC 内部模块
tmc-sdk 依赖 tmc-common
tmc-jedis 依赖 tmc-sdk 和 tmc-common
tmc-server 依赖 tmc-common
tmc-demo 依赖 tmc-jedis
tmc-benchmark 可以依赖 tmc-demo 或独立调用 HTTP 接口
```

禁止依赖：

```text
tmc-common 依赖 tmc-sdk
tmc-sdk 依赖 tmc-jedis
tmc-sdk 依赖 tmc-server
tmc-server 依赖 tmc-sdk
tmc-jedis 依赖 tmc-server
```

## 七、技术栈选型

推荐：

```text
Java 17
```

原因：

```text
LTS 版本
生态稳定
适合 Spring Boot 3.x
JVM 诊断能力完善
```

```yaml
      - --initial-cluster-state=new
```

## 十三、异常规范

公共异常定义在 `tmc-common`。

基础异常：

```java
public class TmcException extends RuntimeException {

    public TmcException(String message) {
        super(message);
    }

    public TmcException(String message, Throwable cause) {
        super(message, cause);
    }
}
```

后续扩展：

```text
TmcConfigException
TmcSerializeException
TmcReportException
TmcEtcdException
TmcCacheException
```

异常处理原则：

```text
启动阶段配置错误可以失败
服务端内部错误可以返回 5xx
SDK 读写链路中的 TMC 异常不能影响 Redis 主链路
Jedis 接入层异常必须尽量降级为原生 Redis 访问
```

## 二十、总结

阶段一是整个 TMC 项目的基础工程阶段。

它不追求业务功能完整，而是确保：

```text
模块边界清晰
依赖方向正确
工程可以编译
本地环境可运行
后续开发有稳定基础
```

只有阶段一做好，后续的 SDK、本地缓存、Jedis 改造、热点探测、etcd 推送、失效广播和压测验证才能顺利推进。
