# Phase 1：工程基础与本地环境设计

## 1. 阶段目标

Phase 1 的目标是建立 TMC 复现项目的工程基础，让后续 SDK、Jedis 接入层、rsyslog + Kafka 采集链路、热点探测服务端、etcd 推送链路都能在一个稳定的本地环境上逐步开发。

这一阶段不实现热点探测算法和业务逻辑，只搭好“能开发、能启动、能验证”的底座。

本阶段必须遵循有赞 TMC 原文的整体思路：

- 应用侧通过 Jedis 风格客户端接入。
- 访问事件通过异步采集链路进入服务端。
- 采集链路保留 rsyslog + Kafka。
- 热点结果和失效事件通过 etcd 推送。
- Redis 作为集中缓存存储。

参考文章：https://tech.youzan.com/tmc/

## 2. 阶段产出

Phase 1 完成后，项目应具备以下能力：

1. Maven 多模块工程结构清晰。
2. 每个模块都有标准源码和测试目录。
3. 本地基础设施可以一条命令启动。
4. Redis、etcd、Kafka、rsyslog 的端口和配置固定。
5. 父工程可以完成编译。
6. README 能说明项目定位、模块职责和本地启动方式。
7. `.gitignore` 能排除 IDE 文件、构建产物、日志和本地运行数据。
8. 后续阶段可以直接在当前工程上写公共模型、SDK 和 server。

## 3. 当前模块定位

当前项目已经拆分为 6 个 Maven 模块，Phase 1 需要确认这些模块的职责和依赖方向。

### 3.1 `tmc-common`

公共模块。

后续承载：

- 事件模型。
- 热点 key 模型。
- 配置模型。
- JSON 工具。
- 常量和协议定义。

依赖要求：

- 不依赖其他 TMC 模块。
- 可以依赖 Jackson、commons-lang3 等通用工具。

### 3.2 `tmc-sdk`

客户端核心 SDK。

后续承载：

- 热点 key 管理。
- 本地缓存。
- 访问事件上报。
- etcd 热点监听。
- etcd 失效广播。
- 客户端指标。

依赖要求：

- 依赖 `tmc-common`。
- 不依赖 `tmc-jedis`。
- 不依赖 `tmc-server`。

### 3.3 `tmc-jedis`

Jedis 接入模块。

后续承载：

- `TmcJedis`。
- `TmcJedisPool`。
- Spring Boot 自动配置。
- Jedis 读写操作拦截。

依赖要求：

- 依赖 `tmc-common`。
- 依赖 `tmc-sdk`。
- 依赖 Jedis。
- 不依赖 `tmc-server`。

### 3.4 `tmc-server`

热点探测服务端。

后续承载：

- Kafka 消费。
- 访问事件聚合。
- 滑动窗口。
- 热点 key 探测。
- etcd 热点快照发布。
- 查询和指标接口。

依赖要求：

- 依赖 `tmc-common`。
- 不依赖 `tmc-sdk`。
- 不依赖 `tmc-jedis`。

### 3.5 `tmc-demo`

业务演示应用。

后续承载：

- Redis 直连接口。
- TMC 读取接口。
- 写入、删除、过期接口。
- 热点访问模拟接口。
- 本地命中率和热点状态展示。

依赖要求：

- 依赖 `tmc-common`。
- 依赖 `tmc-jedis`。
- 依赖 Jedis 和 Spring Boot Web。

### 3.6 `tmc-benchmark`

压测模块。

后续承载：

- direct Redis 压测。
- TMC 模式压测。
- 热点识别耗时统计。
- 本地缓存命中率统计。
- Redis 请求下降比例统计。

依赖要求：

- 依赖 `tmc-common`。
- 可以依赖 Jedis 和 WebClient。
- 不反向依赖业务 demo。

## 5. 本地基础设施规划

Phase 1 需要用 Docker Compose 固定基础设施，后续开发不再手动拼环境。

### 5.1 服务清单

| 服务 | 用途 | 默认端口 |
|---|---|---|
| Redis | 集中缓存存储 | `6379` |
| etcd | 热点 key 下发、失效事件广播 | `2379` |
| Kafka | 访问事件消息队列 | `9092` |
| rsyslog | SDK 访问事件采集入口 | `5514` |

### 5.2 Redis

Redis 在本项目中模拟文章里的缓存集群存储层。

Phase 1 要求：

- 使用单节点 Redis。
- 暴露 `6379`。
- 配置持久化目录，便于本地调试。
- 提供 `redis-cli ping` 验证方式。

验收命令：

```powershell
docker exec tmc-redis redis-cli ping
```

期望输出：

```text
PONG
```

### 5.3 etcd

etcd 在本项目中模拟文章里的基础组件推送能力。

Phase 1 要求：

- 使用单节点 etcd。
- 暴露 `2379`。
- 开启 v3 API。
- 固定 key 前缀规划。

建议 key 前缀：

```text
/tmc/hotkeys/{appName}
/tmc/invalidation/{appName}/{eventId}
/tmc/config/{appName}
```

验收命令：

```powershell
docker exec tmc-etcd etcdctl endpoint health
```

期望结果：

```text
127.0.0.1:2379 is healthy
```

### 5.4 Kafka

Kafka 在本项目中承载访问事件消息。

Phase 1 要求：

- 本地可以使用 KRaft 模式或 ZooKeeper 模式，优先选择配置更简单的单节点模式。
- 暴露 `9092`。
- 创建访问事件 topic。

建议 topic：

```text
tmc-access-events
```

建议 consumer group：

```text
tmc-server
```

验收命令：

```powershell
docker exec tmc-kafka kafka-topics --bootstrap-server localhost:9092 --list
```

期望包含：

```text
tmc-access-events
```

### 5.5 rsyslog

rsyslog 在本项目中承载 SDK 到 Kafka 之间的本地异步采集入口。

Phase 1 要求：

- rsyslog 监听 TCP 或 UDP `5514`。
- 接收 SDK 写入的 JSON line。
- 转发到 Kafka topic `tmc-access-events`。
- 配置文件放在 `infra/rsyslog/rsyslog.conf`。

建议消息格式：

```json
{"protocolVersion":"1.0","appName":"tmc-demo","key":"product:10001","timestamp":1720000000000,"weight":1,"clientId":"demo-1","operation":"GET"}
```

验收方式：

1. 向 rsyslog 端口写入一条 JSON line。
2. 从 Kafka topic 消费到相同 JSON。

## 6. Maven 工程规范

### 6.1 Java 和编码

统一约束：

- Java 17。
- UTF-8。
- Maven 多模块构建。
- JUnit 5。
- Spring Boot 3.2.x。

### 6.2 父工程职责

父 `pom.xml` 负责：

- 管理模块列表。
- 管理依赖版本。
- 管理 Maven 插件版本。
- 管理 Java 编译版本。
- 管理测试插件。

### 6.3 模块构建顺序

预期构建顺序：

```text
tmc-common
  -> tmc-sdk
  -> tmc-jedis
  -> tmc-server
  -> tmc-demo
  -> tmc-benchmark
```

实际 Maven 会按依赖关系构建。Phase 1 需要保证不存在反向依赖和循环依赖。

### 6.4 编译验收

验收命令：

```powershell
mvn -DskipTests compile
```

测试命令：

```powershell
mvn test
```

如果首次下载依赖较慢，可以只要求最终命令能完成，不以首次耗时判断失败。

## 12. 常见问题

### 12.1 Maven 首次编译很慢

原因：

- 首次需要下载 Spring Boot、Jedis、jetcd、Caffeine、测试依赖。

处理方式：

- 保持网络稳定。
- 可以先执行 `mvn -DskipTests compile`。
- 如果下载中断，重新执行即可。

### 12.2 Kafka 本地配置复杂

原因：

- Kafka 对监听地址、advertised listeners、容器内外访问地址敏感。

处理方式：

- Phase 1 固定只支持本机 Docker Compose。
- 先保证容器内 server 可以访问 Kafka。
- demo 和 SDK 后续需要访问时，再补充 host 访问配置。

### 12.3 rsyslog 转 Kafka 插件不可用

原因：

- 不同 rsyslog 镜像可能没有 Kafka 输出模块。

处理方式：

- 选择带 `omkafka` 的镜像。
- 在 `infra/rsyslog/README.md` 中记录镜像选择原因。
- 冒烟脚本必须验证 rsyslog 到 Kafka 的真实链路。

### 12.4 Windows 路径和容器挂载问题

原因：

- Windows 下 Docker 挂载路径和换行符容易影响配置文件。

处理方式：

- 配置文件使用 UTF-8。
- shell 脚本使用 PowerShell。
- 文档中明确从项目根目录执行命令。

## 13. 交付边界

Phase 1 完成后，项目还不会具备热点探测能力，也不会具备真实 TMC 读写拦截能力。

- 能编译。
- 能启动基础设施。
- 能证明 rsyslog 到 Kafka 链路可用。
- 能支撑 Phase 2 开始编写公共模型和协议。
