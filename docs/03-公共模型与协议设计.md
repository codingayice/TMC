# Phase 2：公共模型与协议设计

## 1. 阶段目标

Phase 2 的目标是在 `tmc-common` 中定义 TMC 全链路共享的数据模型、协议常量、key 规则和 JSON 工具。

这一阶段是后续所有模块的基础：

- `tmc-sdk` 需要用 `AccessEvent` 写入 rsyslog。
- `tmc-server` 需要从 Kafka 消费并反序列化 `AccessEvent`。
- `tmc-server` 需要生成 `HotKeySnapshot` 并写入 etcd。
- `tmc-sdk` 需要 watch etcd 并反序列化热点快照。
- `tmc-jedis` 的写操作需要生成 `InvalidationEvent`。
- 多个 SDK 节点需要通过 etcd 交换失效事件。

本阶段只开发 `tmc-common`，不进入 SDK、Jedis 或 server 的业务实现。

## 2. 设计原则

### 2.1 贴合有赞 TMC 原文链路

模型必须服务于原文中的核心流程：

```text
Jedis Client
  -> Hermes/TMC SDK
  -> 访问事件上报
  -> Kafka
  -> Hermes/TMC Server
  -> 3 秒滑动窗口热点探测
  -> etcd 下发热点 key
  -> SDK 本地缓存
  -> 写操作触发失效广播
```

### 2.2 协议字段稳定

事件会跨越多个模块和基础设施：

- Java 对象。
- JSON line。
- rsyslog。
- Kafka message。
- etcd value。

因此字段命名要稳定，避免后续阶段频繁改动。

### 2.3 模型保持简单

Phase 2 的模型只表达协议，不承载业务逻辑。

建议：

- 使用不可变对象或简单 POJO。
- 字段类型优先使用 JDK 基础类型。
- 时间统一使用 epoch millis，即 `long`。
- 操作类型使用枚举。
- 不在模型中直接依赖 Redis、Kafka、etcd 客户端类型。

## 3. 包结构

建议在 `tmc-common` 中建立以下包：

```text
cn.ayice.tmc.common.model
cn.ayice.tmc.common.protocol
cn.ayice.tmc.common.constant
cn.ayice.tmc.common.util
```

### 3.1 `model`

放跨模块传递的数据对象。

建议类：

- `AccessEvent`
- `HotKey`
- `HotKeySnapshot`
- `InvalidationEvent`

### 3.2 `protocol`

放协议版本、topic、etcd key 规则、操作类型。

建议类：

- `EventProtocol`
- `CacheOperation`
- `EtcdKeys`

### 3.3 `constant`

放全局常量。

建议类：

- `TmcConstants`

### 3.4 `util`

放通用工具。

建议类：

- `JsonUtils`

## 4. 核心模型设计

### 4.1 `AccessEvent`

访问事件，表示业务应用读取某个缓存 key。

该事件由 `tmc-sdk` 在读请求路径生成，经过 rsyslog 写入 Kafka，再由 `tmc-server` 消费。

字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `protocolVersion` | `String` | 是 | 协议版本，默认 `1.0` |
| `appName` | `String` | 是 | 业务应用名 |
| `key` | `String` | 是 | 被访问的缓存 key |
| `timestamp` | `long` | 是 | 事件发生时间，epoch millis |
| `weight` | `int` | 是 | 访问权重，默认 `1` |
| `clientId` | `String` | 是 | SDK 实例 ID |
| `operation` | `CacheOperation` | 是 | 初始阶段固定为 `GET` |

JSON 示例：

```json
{
  "protocolVersion": "1.0",
  "appName": "tmc-demo",
  "key": "product:10001",
  "timestamp": 1720000000000,
  "weight": 1,
  "clientId": "demo-1",
  "operation": "GET"
}
```

约束：

- `appName` 不允许为空。
- `key` 不允许为空。
- `weight` 必须大于 `0`。
- `timestamp` 必须大于 `0`。

### 4.2 `HotKey`

热点 key，表示服务端在滑动窗口中识别出的热点。

字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `appName` | `String` | 是 | 业务应用名 |
| `key` | `String` | 是 | 热点缓存 key |
| `score` | `long` | 是 | 当前窗口热度 |
| `detectedAt` | `long` | 是 | 探测时间，epoch millis |
| `ttlMillis` | `long` | 是 | 热点有效期 |

JSON 示例：

```json
{
  "appName": "tmc-demo",
  "key": "product:10001",
  "score": 3580,
  "detectedAt": 1720000003000,
  "ttlMillis": 30000
}
```

约束：

- `score` 必须大于等于 `0`。
- `ttlMillis` 必须大于 `0`。

### 4.3 `HotKeySnapshot`

热点快照，表示某个 app 在一次探测周期中的热点 key 列表。

该对象由 `tmc-server` 写入 etcd，`tmc-sdk` 通过 watch 接收。

字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `protocolVersion` | `String` | 是 | 协议版本 |
| `appName` | `String` | 是 | 业务应用名 |
| `version` | `long` | 是 | 快照版本，单调递增 |
| `hotKeys` | `List<HotKey>` | 是 | 热点 key 列表 |
| `windowSeconds` | `int` | 是 | 探测窗口大小，默认 `30` |
| `publishedAt` | `long` | 是 | 发布时间，epoch millis |

JSON 示例：

```json
{
  "protocolVersion": "1.0",
  "appName": "tmc-demo",
  "version": 12,
  "hotKeys": [
    {
      "appName": "tmc-demo",
      "key": "product:10001",
      "score": 3580,
      "detectedAt": 1720000003000,
      "ttlMillis": 30000
    }
  ],
  "windowSeconds": 30,
  "publishedAt": 1720000003000
}
```

约束：

- `hotKeys` 不能为空集合时表示当前没有热点。
- `version` 必须大于等于 `0`。
- `windowSeconds` 必须大于 `0`。

### 4.4 `InvalidationEvent`

失效事件，表示某个 key 被写入、删除或设置过期后，需要删除本地缓存。

该对象由当前 SDK 节点写入 etcd，其他 SDK 节点通过 watch 接收。

字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `protocolVersion` | `String` | 是 | 协议版本 |
| `appName` | `String` | 是 | 业务应用名 |
| `key` | `String` | 是 | 需要失效的缓存 key |
| `operation` | `CacheOperation` | 是 | `SET`、`DEL`、`EXPIRE` |
| `clientId` | `String` | 是 | 事件发布节点 |
| `eventId` | `String` | 是 | 事件唯一 ID |
| `timestamp` | `long` | 是 | 事件发生时间，epoch millis |

JSON 示例：

```json
{
  "protocolVersion": "1.0",
  "appName": "tmc-demo",
  "key": "product:10001",
  "operation": "SET",
  "clientId": "demo-1",
  "eventId": "demo-1-1720000005000-product-10001",
  "timestamp": 1720000005000
}
```

约束：

- `eventId` 必须唯一。
- SDK 接收到自己发布的 `clientId` 时应忽略该事件。
- `operation` 只能使用会导致缓存值变化的操作。

## 5. 协议常量设计

### 5.1 `EventProtocol`

建议职责：

- 定义当前协议版本。
- 提供默认方法判断协议版本是否兼容。

建议常量：

```java
CURRENT_VERSION = "1.0"
```

版本兼容策略：

- Phase 2 只支持 `1.0`。
- 反序列化时允许读取未知字段，保证后续协议扩展。

### 5.2 `CacheOperation`

缓存操作类型。

建议枚举值：

```text
GET
MGET
SET
DEL
EXPIRE
```

用途：

- `GET`、`MGET` 用于访问事件。
- `SET`、`DEL`、`EXPIRE` 用于失效事件。

建议提供方法：

- `isReadOperation()`
- `isWriteOperation()`
- `requiresInvalidation()`

### 5.3 `TmcConstants`

建议常量：

```text
DEFAULT_ACCESS_EVENT_TOPIC = "tmc-access-events"
DEFAULT_WINDOW_SECONDS = 30
DEFAULT_BUCKET_SECONDS = 3
DEFAULT_BUCKET_COUNT = 10
DEFAULT_HOT_KEY_TTL_MILLIS = 30000
DEFAULT_EVENT_WEIGHT = 1
```

## 6. etcd key 规则

### 6.1 设计目标

etcd key 需要支持：

- 按应用隔离热点 key。
- 按应用隔离失效事件。
- SDK 可以 watch 一个稳定前缀。
- key 中不能出现不安全字符导致路径歧义。

### 6.2 建议路径

热点快照：

```text
/tmc/hotkeys/{appName}
```

失效事件：

```text
/tmc/invalidation/{appName}/{eventId}
```

应用配置：

```text
/tmc/config/{appName}
```

### 6.3 `EtcdKeys`

建议方法：

```java
hotKeysPath(String appName)
invalidationPrefix(String appName)
invalidationPath(String appName, String eventId)
configPath(String appName)
```

路径规范：

- 所有路径以 `/tmc` 开头。
- `appName` 不允许为空。
- `eventId` 不允许为空。
- `appName` 和 `eventId` 中的 `/` 需要被拒绝或转义。

Phase 2 建议直接拒绝包含 `/` 的 path segment，降低复杂度。

## 7. JSON 工具设计

### 7.1 `JsonUtils`

建议使用 Jackson，并在 `tmc-common` 内集中封装。

建议方法：

```java
toJson(Object value)
fromJson(String json, Class<T> type)
toJsonLine(Object value)
```

要求：

- `toJsonLine` 输出末尾带 `\n`，服务于 rsyslog JSON line。
- 反序列化时忽略未知字段。
- 不输出 `null` 字段，降低 Kafka 消息体积。
- 枚举使用字符串，例如 `"GET"`。

### 7.2 ObjectMapper 配置

建议配置：

- `FAIL_ON_UNKNOWN_PROPERTIES = false`
- `Include.NON_NULL`
- 枚举按字符串序列化

## 8. 测试设计

Phase 2 必须补 `tmc-common` 单元测试。

建议测试类：

```text
AccessEventTest
HotKeySnapshotTest
InvalidationEventTest
EtcdKeysTest
JsonUtilsTest
CacheOperationTest
```

### 8.1 JSON round-trip 测试

每个事件对象都要覆盖：

1. 构造对象。
2. 序列化成 JSON。
3. 反序列化回对象。
4. 断言关键字段一致。

覆盖对象：

- `AccessEvent`
- `HotKey`
- `HotKeySnapshot`
- `InvalidationEvent`

### 8.2 JSON line 测试

验证：

- `toJsonLine` 末尾有且只有一个换行。
- 去掉换行后仍然是合法 JSON。
- 适合写入 rsyslog。

### 8.3 etcd key 测试

验证：

- `hotKeysPath("tmc-demo")` 返回 `/tmc/hotkeys/tmc-demo`。
- `invalidationPrefix("tmc-demo")` 返回 `/tmc/invalidation/tmc-demo/`。
- `invalidationPath("tmc-demo", "event-1")` 返回 `/tmc/invalidation/tmc-demo/event-1`。
- appName 为空时报错。
- appName 包含 `/` 时报错。

### 8.4 操作类型测试

验证：

- `GET` 和 `MGET` 是读操作。
- `SET`、`DEL`、`EXPIRE` 是写操作。
- `SET`、`DEL`、`EXPIRE` 需要触发失效。
- `GET`、`MGET` 不触发失效。

## 11. 和后续阶段的关系

Phase 2 完成后，Phase 3 可以直接使用：

- `AccessEvent`：SDK 读请求生成访问事件。
- `HotKey`：SDK 维护热点 key 表。
- `HotKeySnapshot`：SDK 接收服务端热点快照。
- `InvalidationEvent`：SDK 发布和接收失效事件。
- `JsonUtils`：SDK 写 rsyslog，server 读 Kafka。
- `EtcdKeys`：SDK 和 server 对齐 etcd 路径。
- `CacheOperation`：Jedis 接入层区分读写操作。

Phase 2 的质量会直接影响后续所有模块，所以这里宁愿多写测试，也不要把协议问题留到 SDK 和 server 阶段再修。
