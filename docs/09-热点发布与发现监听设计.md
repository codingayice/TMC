# Phase 8：热点发布与发现监听设计

## 1. 阶段目标

Phase 8 的目标是打通服务端热点发现结果到 SDK 的下发链路。

Phase 7 已经实现：

```text
Kafka AccessEvent
  -> tmc-server
  -> 内存访问热度累加
  -> 时间轮滑动窗口
  -> 热点探测
  -> HotKeySnapshot
```

本阶段要在这个基础上继续向后推进：

```text
HotKeySnapshot
  -> tmc-server 发布到 etcd
  -> tmc-sdk 监听 etcd
  -> HotKeyManager 更新本地热点 key 集合
  -> TmcClient 热点读请求进入本地缓存路径
```

完成后应具备以下能力：

1. `tmc-server` 可以把每个应用最新的 `HotKeySnapshot` 写入 etcd。
2. `tmc-server` 可以在热点消失时发布空快照或删除 etcd key，避免 SDK 保留旧热点。
3. `tmc-sdk` 启动时可以从 etcd 拉取当前应用的热点快照。
4. `tmc-sdk` 运行中可以 watch 当前应用的热点快照路径。
5. SDK 收到热点快照后调用 `HotKeyManager.updateHotKeySnapshot(...)` 替换本地热点集合。
6. SDK 收到热点快照删除事件后清空本地热点集合。
7. etcd 通信异常不会影响业务 Redis 读写主流程。
8. 单元测试覆盖热点发布、热点监听、快照替换、异常隔离和自动配置。

## 2. 原文思路对齐

有赞原文中，热点 key 识别后会通过配置中心下发到业务应用，业务应用 SDK 监听热点变化并更新本地缓存策略。

本项目使用 etcd 复现这个下发过程：

| 原文能力 | 本项目实现 |
|---|---|
| 服务端识别热点 key | `HotKeyDetector` 生成 `HotKeySnapshot` |
| 热点 key 结果写入配置中心 | `HotKeyPublisher` 写入 etcd |
| SDK 监听热点 key 变化 | `HotKeyDiscoveryListener` watch etcd |
| SDK 更新本地热点判断表 | `HotKeyManager.updateHotKeySnapshot(...)` |
| 热点 key 读请求使用本地缓存 | `TmcClient.get(...)` 在 `hotKeyManager.isHotKey(key)` 为 true 时访问 `CaffeineLocalCache` |

本阶段只实现热点发现监听，不实现缓存失效广播。缓存失效广播属于后续阶段。

## 3. 阶段边界

本阶段实现：

- 服务端 etcd 连接配置。
- 服务端热点快照发布。
- 服务端热点快照删除或空快照发布。
- SDK etcd 连接配置。
- SDK 启动时拉取热点快照。
- SDK 运行时监听热点快照变化。
- SDK 热点快照版本去重。
- SDK 通信线程和业务线程隔离。
- SDK 和 server 的指标补充。
- 单元测试。

本阶段不实现：

- 写操作失效事件上报。
- 热点失效监听。
- 跨节点本地缓存失效广播。
- 管理后台、控制台和人工配置入口。

这些能力会在后续阶段单独设计。

## 4. etcd Key 设计

热点快照使用已有 `EtcdKeys.hotKeysPath(appName)` 生成路径。

```text
/tmc/hotkeys/{appName}
```

示例：

```text
/tmc/hotkeys/product-service
```

设计原则：

- 一个应用只保留一个热点快照 key。
- 服务端每次探测后覆盖写入最新快照。
- SDK 只监听自己 `appName` 对应的 key，不监听其他应用。
- 热点快照不需要 `eventId`，因为它表达的是“当前最新状态”，不是“每次独立事件”。
- 失效事件后续会使用 `/tmc/invalidation-events/{appName}/{eventId}`，因为每次写操作都必须被独立消费，不能互相覆盖。

## 5. 数据协议

etcd value 使用 `HotKeySnapshot` 的 JSON 字符串。

示例：

```json
{
  "version": "product-service-1720000003000",
  "appName": "product-service",
  "hotKeys": [
    {
      "appName": "product-service",
      "key": "product:10001",
      "score": 120,
      "detectedAt": 1720000003000,
      "ttlMillis": 300000
    }
  ]
}
```

空热点快照示例：

```json
{
  "version": "product-service-1720000006000",
  "appName": "product-service",
  "hotKeys": []
}
```

空快照的语义：

```text
当前 app 没有达到阈值的热点 key，SDK 应清空本地热点集合。
```

删除 key 的语义：

```text
服务端停止追踪该 app，或热点快照过期，SDK 应清空本地热点集合。
```

## 6. 模块结构

本阶段涉及 `tmc-server` 和 `tmc-sdk`。

### 6.1 `tmc-server`

建议新增或修改：

```text
cn.ayice.tmc.server.config.EtcdProperties
cn.ayice.tmc.server.config.TmcServerProperties
cn.ayice.tmc.server.config.TmcServerConfiguration
cn.ayice.tmc.server.hotkey.HotKeyPublisher
cn.ayice.tmc.server.hotkey.HotKeyMappingScheduler
cn.ayice.tmc.server.metrics.TmcServerMetrics
```

职责说明：

| 类 | 职责 |
|---|---|
| `EtcdProperties` | 保存服务端 etcd 地址、连接超时、热点快照租约 TTL 等配置 |
| `TmcServerProperties` | 挂载 `etcd` 配置分组 |
| `TmcServerConfiguration` | 创建 jetcd `Client` 和 `HotKeyPublisher` |

| `HotKeyMappingScheduler` | 探测出快照后调用 `HotKeyPublisher` |
| `TmcServerMetrics` | 增加发布成功、发布失败、删除成功、删除失败指标 |

说明：

- `HotKeyPublisher` 是 concrete class，不额外抽 Java `interface`。
- 本项目当前只支持 etcd 作为热点发布通道，不设计多实现扩展层。

### 6.2 `tmc-sdk`

建议新增或修改：

```text
cn.ayice.tmc.communication.EtcdProperties

cn.ayice.tmc.sdk.TmcAutoConfiguration
cn.ayice.tmc.sdk.TmcMetrics

```

职责说明：

| 类 | 职责 |
|---|---|
| `EtcdProperties` | 保存 SDK 连接 etcd 的地址、超时和重试参数 |
| `HotKeyDiscoveryProperties` | 保存热点发现监听开关、启动拉取开关、watch 线程参数 |
| `HotKeyDiscoveryListener` | 启动时拉取快照，运行时 watch 快照变化，并更新 `HotKeyManager` |
| `HotKeyManager` | 增加清空热点集合能力，继续负责本地热点状态维护 |
| `TmcAutoConfiguration` | 在配置开启时自动创建 `HotKeyDiscoveryListener` |
| `TmcMetrics` | 增加热点发现监听相关指标 |
| `TmcProperties` | 挂载 `etcd` 和 `hot-key.discovery` 配置 |

说明：

- `HotKeyDiscoveryListener` 属于通信模块。
- `HotKeyManager` 仍然属于热点模块，只管理内存状态，不直接依赖 etcd。
- `TmcClient` 不需要感知 etcd，只继续依赖 `HotKeyManager`。

## 7. 配置设计

### 7.1 服务端配置

```yaml
tmc:
  server:
    etcd:
      endpoints:
        - http://localhost:2379
      connect-timeout-millis: 3000
      hot-key-lease-ttl-seconds: 30
      enabled: true
```

字段说明：

| 字段 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `enabled` | `boolean` | `true` | 是否启用热点快照发布 |
| `endpoints` | `List<String>` | `http://localhost:2379` | etcd 地址列表 |
| `connectTimeoutMillis` | `long` | `3000` | 连接超时时间 |
| `hotKeyLeaseTtlSeconds` | `long` | `30` | 热点快照 key 的 etcd lease TTL |

约束：

- `endpoints` 不能为空。
- `connectTimeoutMillis > 0`。
- `hotKeyLeaseTtlSeconds > 0`。
- `hotKeyLeaseTtlSeconds` 建议大于服务端映射周期，避免下一轮发布前 key 提前过期。

### 7.2 SDK 配置

```yaml
tmc:
  etcd:
    endpoints:
      - http://localhost:2379
    connect-timeout-millis: 3000
  hot-key:
    ttl-millis: 300000
    discovery:
      enabled: true
      load-on-startup: true
      reconnect-delay-millis: 3000
```

字段说明：

| 字段 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `tmc.etcd.endpoints` | `List<String>` | `http://localhost:2379` | SDK 连接的 etcd 地址 |
| `tmc.etcd.connectTimeoutMillis` | `long` | `3000` | etcd 连接超时时间 |
| `tmc.hot-key.discovery.enabled` | `boolean` | `true` | 是否启用热点发现监听 |
| `tmc.hot-key.discovery.loadOnStartup` | `boolean` | `true` | 启动时是否先拉取一次当前快照 |
| `tmc.hot-key.discovery.reconnectDelayMillis` | `long` | `3000` | watch 异常后的重连等待时间 |

约束：

- `tmc.app-name` 仍然是必填配置。
- `tmc.etcd.endpoints` 不能为空。
- `reconnectDelayMillis > 0`。

## 8. 服务端热点发布设计

### 8.1 `HotKeyPublisher`

职责：

- 将 `HotKeySnapshot` 序列化为 JSON。
- 写入 `EtcdKeys.hotKeysPath(snapshot.getAppName())`。
- 写入时绑定 lease，让服务端异常退出后热点快照可以自动过期。
- 在没有可发布快照时删除对应 app 的热点快照 key。
- 捕获发布异常并交给指标记录，不能影响下一轮映射任务。

建议方法：

```java
public class HotKeyPublisher implements AutoCloseable {
    public void publish(HotKeySnapshot snapshot) {
    }

    public void delete(String appName) {
    }

    @Override
    public void close() {
    }
}
```

发布流程：

```text
HotKeySnapshot
  -> validate appName
  -> JsonUtils.toJson(snapshot)
  -> lease.grant(hotKeyLeaseTtlSeconds)
  -> kv.put(EtcdKeys.hotKeysPath(appName), json, leaseId)
  -> metrics.incrementHotKeyPublishSuccess()
```

删除流程：

```text
appName
  -> EtcdKeys.hotKeysPath(appName)
  -> kv.delete(path)
  -> metrics.incrementHotKeyDeleteSuccess()
```

异常处理：

```text
publish/delete 失败
  -> metrics increment failed
  -> 记录 warn 日志
  -> 返回，不抛到映射主流程
```

### 8.2 `HotKeyMappingScheduler` 集成

当前 `HotKeyMappingScheduler` 已经负责生成并保存 `HotKeySnapshot`。

本阶段修改后流程：

```text
mapApp(appName)
  -> drain 当前周期访问热度
  -> 推进时间轮
  -> 清理冷 key
  -> detector.detect(appName, wheelMap)
  -> repository.save(snapshot)
  -> publisher.publish(snapshot)
```

当 `wheelMap` 为空：

```text
wheelStore.remove(appName)
repository.remove(appName)
publisher.delete(appName)
```

当 `wheelMap` 不为空但没有热点 key 达到阈值：

```text
snapshot.hotKeys = []
repository.save(snapshot)
publisher.publish(snapshot)
```

这样 SDK 能收到空列表并清空本地热点集合，避免旧热点残留。

## 9. SDK 热点发现监听设计

### 9.1 `HotKeyDiscoveryListener`

职责：

- 根据 `tmc.app-name` 生成监听路径。
- 启动时读取一次当前快照。
- watch 当前应用热点快照 key。
- 收到 PUT 事件后解析 `HotKeySnapshot`。
- 快照 `appName` 不匹配时忽略。
- 快照 `version` 与本地最近版本相同时忽略。
- 快照合法时调用 `HotKeyManager.updateHotKeySnapshot(snapshot)`。
- 收到 DELETE 事件后清空本地热点集合。
- watch 异常时在通信线程内重试，不影响业务线程。

建议方法：

```java
public class HotKeyDiscoveryListener implements AutoCloseable {
    public void start() {
    }

    @Override
    public void close() {
    }
}
```

启动流程：

```text
start()
  -> 如果已启动，直接返回
  -> 如果 loadOnStartup=true，读取 EtcdKeys.hotKeysPath(appName)
  -> 解析当前快照并更新 HotKeyManager
  -> 在线程池中启动 watch
```

PUT 事件流程：

```text
watch event PUT
  -> value bytes 转字符串
  -> JsonUtils.fromJson(value, HotKeySnapshot.class)
  -> 校验 appName
  -> 校验 version 是否重复
  -> hotKeyManager.updateHotKeySnapshot(snapshot)

```

DELETE 事件流程：

```text
watch event DELETE
  -> hotKeyManager.clearHotKeys()
  -> 清空 lastVersion

```

异常处理：

```text
JSON 解析失败

  -> 打 warn 日志
  -> 继续 watch

watch 断开

  -> sleep reconnectDelayMillis
  -> 重新 watch
```

### 9.2 `HotKeyManager` 补充

当前 `HotKeyManager.updateHotKeySnapshot(snapshot)` 已经会先清空本地热点集合，再应用快照中的热点 key。

本阶段建议补充：

```java
public void clearHotKeys() {
    hotKeys.clear();
}
```

原因：

- etcd DELETE 事件没有 value，无法直接调用 `updateHotKeySnapshot(...)`。
- 单独的清空方法语义更直接。
- `HotKeyDiscoveryListener` 不需要伪造一个空 `HotKeySnapshot`。

## 10. 线程隔离设计

热点发现监听属于通信模块，不能占用业务请求线程。

建议：

```text
HotKeyDiscoveryListener
  -> 单独 ScheduledExecutorService 或 ExecutorService
  -> 线程名 tmc-hotkey-discovery
  -> daemon = true
```

线程职责：

- 启动拉取快照。
- 建立 etcd watch。
- 处理 watch 事件。
- 异常后等待并重连。

业务线程职责：

- 执行 `TmcClient.get(...)`。
- 查询 `HotKeyManager.isHotKey(key)`。
- 读取本地缓存或回调 Jedis。

两者只通过 `HotKeyManager` 的并发 map 交互。

## 11. 自动配置设计

`TmcAutoConfiguration` 新增 `HotKeyDiscoveryListener` Bean。

创建条件：

```text
tmc.enabled=true
tmc.hot-key.enabled=true
tmc.hot-key.discovery.enabled=true
存在 HotKeyManager
不存在用户自定义 HotKeyDiscoveryListener
```

建议：

```java
@Bean
@ConditionalOnMissingBean
@ConditionalOnBean(HotKeyManager.class)
@ConditionalOnProperty(
        prefix = "tmc.hot-key.discovery",
        name = "enabled",
        havingValue = "true",
        matchIfMissing = true
)
public HotKeyDiscoveryListener hotKeyDiscoveryListener(...) {
}
```

启动方式：

- `HotKeyDiscoveryListener` 可以实现 `SmartLifecycle`，由 Spring 自动调用 `start()` 和 `stop()`。
- 如果保持简单，也可以在构造后通过 `@PostConstruct` 启动，并在 `close()` 中释放资源。

本项目建议优先使用 `SmartLifecycle`：

- 生命周期语义清晰。
- 便于测试启动和停止。
- 应用关闭时能优雅关闭 watch 和线程池。

## 12. 指标设计

### 12.1 服务端指标

`TmcServerMetrics` 建议新增：

| 指标 | 含义 |
|---|---|
| `hotKeyPublishSuccess` | 热点快照发布成功次数 |
| `hotKeyPublishFailed` | 热点快照发布失败次数 |
| `hotKeyDeleteSuccess` | 热点快照删除成功次数 |
| `hotKeyDeleteFailed` | 热点快照删除失败次数 |

### 12.2 SDK 指标

`TmcMetrics` 建议新增：

| 指标 | 含义 |
|---|---|
| `hotKeySnapshotLoaded` | 启动时成功加载快照次数 |
| `hotKeySnapshotApplied` | watch 收到并成功应用快照次数 |
| `hotKeySnapshotInvalid` | 快照 JSON 或字段非法次数 |
| `hotKeySnapshotDeleted` | 收到删除事件并清空热点次数 |
| `hotKeyWatchReconnect` | watch 异常后重连次数 |
| `hotKeyWatchFailed` | watch 启动或处理失败次数 |

这些指标用于判断：

- 服务端是否持续发布热点结果。
- SDK 是否真的收到热点变更。
- SDK 是否因为 JSON 协议不一致而无法应用快照。
- etcd watch 是否频繁断开。

## 13. 测试设计

### 13.1 `HotKeyPublisherTest`

覆盖：

1. `publish(snapshot)` 会写入 `EtcdKeys.hotKeysPath(appName)`。
2. 写入 value 是 `HotKeySnapshot` JSON。
3. `publish(snapshot)` 会绑定 lease。
4. `delete(appName)` 会删除热点路径。
5. `publish(null)` 不写入 etcd，并记录失败或直接忽略。
6. etcd 写入异常不会向外抛出。

### 13.2 `HotKeyMappingSchedulerTest`

在现有测试基础上补充：

1. 探测出热点快照后会调用 `HotKeyPublisher.publish(snapshot)`。
2. 没有热点但仍有追踪 key 时，会发布空热点快照。
3. `wheelMap` 清空时会调用 `HotKeyPublisher.delete(appName)`。
4. publisher 抛异常时，映射任务不会中断。

### 13.3 `HotKeyDiscoveryListenerTest`

覆盖：

1. 启动时 `loadOnStartup=true` 会读取当前快照。
2. 读取到合法快照会调用 `HotKeyManager.updateHotKeySnapshot(...)`。
3. 收到 PUT 事件会应用新快照。
4. 收到相同 `version` 的 PUT 事件会忽略。
5. 收到其他 `appName` 的快照会忽略。
6. 收到非法 JSON 会记录 invalid 指标，不影响后续监听。
7. 收到 DELETE 事件会调用 `HotKeyManager.clearHotKeys()`。
8. `close()` 会关闭 watch、etcd client 和线程池。

### 13.4 `TmcAutoConfigurationTest`

覆盖：

1. 默认配置下会创建 `HotKeyDiscoveryListener`。
2. `tmc.hot-key.discovery.enabled=false` 时不会创建监听器。
3. 用户自定义 `HotKeyDiscoveryListener` 时，自动配置不覆盖。
4. `tmc.enabled=false` 时不会创建监听器。

### 13.5 `HotKeyManagerTest`

补充：

1. `clearHotKeys()` 会清空所有热点 key。
2. 清空后 `hotKeyCount()` 返回 0。
3. 清空后 `isHotKey(oldKey)` 返回 false。

## 16. 联调验证

本阶段完成后，可以按以下方式验证链路：

1. 启动 Redis、Kafka、rsyslog 和 etcd。
2. 启动 `tmc-server`，配置 Kafka 和 etcd 地址。
3. 启动接入 `tmc-sdk` 和 `tmc-jedis` 的 demo 应用，配置相同 etcd 地址和同一个 `appName`。
4. 通过 demo 对同一个 key 发起高频 `get` 请求。
5. 等待服务端达到热点阈值并发布快照。
6. 使用 `etcdctl get /tmc/hotkeys/{appName}` 查看快照是否存在。
7. 观察 demo SDK 指标，确认热点快照被应用。
8. 再次读取同一个热点 key，确认 `TmcClient` 开始进入本地缓存路径。

完整链路：

```text
TmcJedis.get(key)
  -> TmcClient.get(key, jedis::get)
  -> AccessReporter 写 rsyslog
  -> rsyslog 转发 Kafka
  -> tmc-server 消费 Kafka
  -> 时间轮识别 HotKeySnapshot
  -> HotKeyPublisher 写 etcd
  -> HotKeyDiscoveryListener watch etcd
  -> HotKeyManager 更新热点集合
  -> TmcClient 后续读热点 key 命中本地缓存
```

## 17. 风险与处理

| 风险 | 处理方式 |
|---|---|
| 服务端发布失败 | 记录指标和 warn 日志，下一轮映射继续发布 |
| etcd key 提前过期 | `hotKeyLeaseTtlSeconds` 设置为映射周期的多倍 |
| SDK 启动时 watch 之前已有热点 | `loadOnStartup=true` 先读取一次当前快照 |
| SDK 收到旧快照 | 使用 `version` 去重 |
| SDK 收到其他应用快照 | 校验 `snapshot.appName` |
| JSON 协议不匹配 | 记录 invalid 指标，继续监听 |
| watch 断开 | 通信线程等待 `reconnectDelayMillis` 后重连 |
| 通信异常影响业务请求 | etcd 操作只在监听线程执行，业务线程只读 `HotKeyManager` |
