# Phase 7：服务端热点发现设计

## 1. 阶段目标

Phase 7 的目标是在 `tmc-server` 中实现服务端热点发现能力。

Phase 6 已经验证访问事件可以进入 Kafka：

```text
SDK / TmcJedis
  -> AccessReporter
  -> rsyslog
  -> Kafka topic: tmc-access-events
```

本阶段要让 `tmc-server` 消费 Kafka 中的 `AccessEvent`，按照应用和 key 维度统计访问热度，并基于时间轮滑动窗口识别热点 key。

完成后应具备以下能力：

1. `tmc-server` 可以订阅 `tmc-access-events`。
2. Kafka 消息可以反序列化为 `AccessEvent`。
3. 非法消息不会导致消费主流程退出。
4. 服务端按 `appName + key` 在内存中累加访问热度。
5. 服务端每 3 秒执行一次映射任务，把最近访问热度写入时间轮。
6. 每个 key 维护 10 个时间片，每个时间片代表 3 秒访问量。
7. 10 个时间片累加得到最近 30 秒滑动窗口热度。
8. 服务端可以按应用输出 TopN 热点 key。
9. 热点 key 达到阈值后生成 `HotKeySnapshot`。
10. 单元测试覆盖事件消费、热度累加、时间轮滑窗和热点探测。

本阶段完成后的链路：

```text
Kafka AccessEvent
  -> tmc-server
  -> 内存访问热度累加
  -> 时间轮滑动窗口
  -> 热点探测
  -> HotKeySnapshot
```

## 2. 原文思路对齐

有赞原文中，TMC 热点发现流程分为四步：

```text
数据收集
  -> 热度滑窗
  -> 热度汇聚
  -> 热点探测
```

本项目按同样思路复现：

| 原文能力 | 本项目实现 |
|---|---|
| Hermes-SDK 通过 rsyslog + Kafka 上报 key 访问事件 | SDK `AccessReporter` 写 rsyslog，rsyslog 转发到 `tmc-access-events` |
| Hermes 服务端消费 Kafka 消息 | `AccessEventConsumer` 消费 Kafka |
| `Map<String, Map<String, LongAdder>>` 存储访问热度 | `AccessEventAccumulator` 按 `appName -> key -> LongAdder` 累加 |
| 每个 App 每 3 秒执行映射任务 | `HotKeyMappingScheduler` 定时触发 app 级映射 |
| 每个 key 维护 10 个 3 秒时间片 | `TimeWheelCounter` 维护 10 个 bucket |
| 30 秒滑动窗口总热度 | `TimeWheelCounter.sum()` 汇总 10 个 bucket |
| TopN 热点 key 探测 | `HotKeyDetector` 按阈值和 TopN 生成 `HotKeySnapshot` |

## 3. 阶段边界

本阶段实现：

## 5. 配置设计

### 5.1 配置示例

```yaml
tmc:
  server:
    kafka:
      bootstrap-servers: localhost:9092
      topic: tmc-access-events
      group-id: tmc-hotkey-server
      concurrency: 1
    hot-key:
      window-seconds: 30
      bucket-seconds: 3
      threshold: 100
      top-n: 100
      ttl-millis: 300000
      mapping-interval-millis: 3000
```

### 5.2 `TmcServerProperties`

```java
public class TmcServerProperties {
    private KafkaConsumerProperties kafka = new KafkaConsumerProperties();
    private HotKeyDetectProperties hotKey = new HotKeyDetectProperties();
}
```

### 5.3 `KafkaConsumerProperties`

| 字段 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `bootstrapServers` | `String` | `localhost:9092` | Kafka broker 地址 |
| `topic` | `String` | `tmc-access-events` | 访问事件 topic |
| `groupId` | `String` | `tmc-hotkey-server` | 消费组 |
| `concurrency` | `int` | `1` | Spring Kafka listener 并发数 |

### 5.4 `HotKeyDetectProperties`

| 字段 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `windowSeconds` | `int` | `30` | 滑动窗口长度 |
| `bucketSeconds` | `int` | `3` | 单个时间片长度 |
| `threshold` | `long` | `100` | 热点 key 最小热度阈值 |
| `topN` | `int` | `100` | 每个应用最多输出热点 key 数量 |
| `ttlMillis` | `long` | `300000` | 热点 key 有效期 |
| `mappingIntervalMillis` | `long` | `3000` | 映射任务执行周期 |

约束：

- `windowSeconds > 0`。
- `bucketSeconds > 0`。
- `windowSeconds % bucketSeconds == 0`。
- `threshold > 0`。
- `topN > 0`。
- `ttlMillis > 0`。
- `mappingIntervalMillis > 0`。

## 6. 数据结构设计

### 6.1 当前周期访问热度

服务端消费到 Kafka 消息后，先写入当前周期访问热度表：

```text
Map<appName, Map<key, LongAdder>>
```

业务含义：

```text
Map<业务应用, Map<缓存 key, 当前周期访问热度>>
```

建议在代码中使用：

```java
ConcurrentHashMap<String, ConcurrentHashMap<String, LongAdder>>
```

累加逻辑：

```text
AccessEvent(appName, key, weight)
  -> 找到 appName 对应 key 表
  -> 找到 key 对应 LongAdder
  -> add(weight)
```

### 6.2 时间轮滑窗

每个 `appName + key` 维护一个时间轮：

```text
TimeWheelCounter
  -> long[] buckets = new long[10]
  -> currentIndex
  -> lastTick
```

如果配置是：

```text
windowSeconds = 30
bucketSeconds = 3
```

则：

```text
bucketCount = 30 / 3 = 10
```

每次映射任务执行时，把当前周期中该 key 的访问热度写入当前时间片：

```text
bucket[currentIndex] = currentPeriodScore
currentIndex = (currentIndex + 1) % bucketCount
```

当前 key 的 30 秒滑窗热度为：

```text
sum(bucket[0..9])
```

### 6.3 App 级时间轮表

建议结构：

```text
Map<appName, Map<key, TimeWheelCounter>>
```

业务含义：

```text
Map<业务应用, Map<缓存 key, key 对应的时间轮>>
```

## 7. 核心组件设计

### 7.1 `AccessEventConsumer`

职责：

- 订阅 Kafka topic `tmc-access-events`。
- 接收 JSON line。
- 使用 `JsonUtils.fromJson(message, AccessEvent.class)` 反序列化。
- 校验必要字段。
- 将合法事件交给 `AccessEventAccumulator`。
- 记录消费成功、消费失败、非法消息指标。

处理流程：

```text
Kafka message
  -> JsonUtils.fromJson
  -> validate AccessEvent
  -> accumulator.add(event)
```

字段校验：

| 字段 | 规则 |
|---|---|
| `appName` | 非空 |
| `key` | 非空 |
| `timestamp` | 非空且大于 0 |
| `weight` | 大于 0 |
| `operation` | 当前接受 `GET` |

异常处理：

```text
JSON 反序列化失败
  -> invalidMessages + 1
  -> 打 warn 日志
  -> 返回

字段非法
  -> invalidMessages + 1
  -> 打 warn 日志
  -> 返回

accumulator 异常
  -> consumeFailed + 1
  -> 打 error 日志
  -> 返回
```

消费失败不能让 listener 线程因为单条坏消息停止。

### 7.2 `AccessEventAccumulator`

职责：

- 保存当前周期内每个 app/key 的访问热度。
- 提供线程安全的 `add(AccessEvent event)`。
- 提供 `drainApp(String appName)`，供映射任务取出当前周期热度并清空。
- 提供 `appNames()`，供调度器知道当前有哪些 app。

建议方法：

```java
public class AccessEventAccumulator {
    public void add(AccessEvent event) {
    }

    public Set<String> appNames() {
    }

    public Map<String, Long> drainApp(String appName) {
    }
}
```

`drainApp(appName)` 行为：

```text
找到 appName 对应 key 表
  -> 原子替换为空表
  -> 将 LongAdder.sum() 转成 Map<key, score>
  -> 返回
```

这样消费线程可以继续写新表，映射线程处理旧表，减少锁竞争。

### 7.3 `TimeWheelCounter`

职责：

- 保存一个 key 最近 30 秒的访问热度。
- 每次映射任务写入当前 3 秒时间片。
- 提供 `sum()` 返回滑窗总热度。

建议方法：

```java
public class TimeWheelCounter {
    public TimeWheelCounter(int bucketCount) {
    }

    public void addBucket(long score) {
    }

    public long sum() {
    }
}
```

行为：

```text
addBucket(score)
  -> buckets[currentIndex] = score
  -> currentIndex = (currentIndex + 1) % bucketCount

sum()
  -> 累加所有 bucket
```

注意：

- `addBucket` 是覆盖当前时间片，不是累加当前时间片。
- 当某个 key 当前周期没有访问时，如果它仍在时间轮表中，需要写入 `0`，否则旧热度会长期残留。

### 7.4 `HotKeyMappingScheduler`

职责：

- 每 `mappingIntervalMillis` 触发一次映射任务。
- 对每个 app 执行一次映射。
- 将当前周期热度写入对应 key 的时间轮。
- 对已有时间轮但当前周期没有访问的 key 写入 `0`。
- 调用 `HotKeyDetector` 生成最新热点快照。
- 将结果保存到 `HotKeySnapshotRepository`。

映射流程：

```text
for appName in accumulator.appNames() + knownApps
  currentScores = accumulator.drainApp(appName)
  wheelMap = wheelStore.get(appName)

  for key in currentScores
    wheelMap.computeIfAbsent(key)
    wheel.addBucket(score)

  for key in wheelMap but not in currentScores
    wheel.addBucket(0)

  snapshot = detector.detect(appName, wheelMap)
  repository.save(snapshot)
```

`knownApps` 的作用：

- 某个 app 当前周期没有访问事件时，也要继续推动它已有 key 的时间轮向前滑动。
- 否则旧热点会因为没有新事件而停留在窗口里。

### 7.5 `HotKeyDetector`

职责：

- 汇总每个 key 的时间轮总热度。
- 过滤低于 `threshold` 的 key。
- 按热度倒序排序。
- 截取 TopN。
- 生成 `HotKeySnapshot`。

探测流程：

```text
Map<key, TimeWheelCounter>
  -> key + wheel.sum()
  -> score >= threshold
  -> sort by score desc
  -> limit topN
  -> HotKey
  -> HotKeySnapshot
```

`HotKey` 字段：

| 字段 | 来源 |
|---|---|
| `appName` | 当前 app |
| `key` | 当前 key |
| `score` | 时间轮 30 秒总热度 |
| `detectedAt` | 当前系统时间 |
| `ttlMillis` | `HotKeyDetectProperties.ttlMillis` |

`HotKeySnapshot` 字段：

| 字段 | 来源 |
|---|---|
| `version` | `appName + timestamp` 或 UUID |
| `appName` | 当前 app |
| `hotKeys` | TopN 热点 key 列表 |

### 7.6 `HotKeySnapshotRepository`

职责：

- 保存每个 app 最近一次热点快照。
- 提供查询方法，方便单元测试、调试接口和后续发布流程使用。

建议方法：

```java
public class HotKeySnapshotRepository {
    public void save(HotKeySnapshot snapshot) {
    }

    public HotKeySnapshot get(String appName) {
    }

    public Map<String, HotKeySnapshot> getAll() {
    }
}
```

## 8. Kafka 消费设计

`tmc-server` 使用 Spring Boot，建议接入 Spring Kafka。

`tmc-server/pom.xml` 增加：

```xml
<dependency>
    <groupId>org.springframework.kafka</groupId>
    <artifactId>spring-kafka</artifactId>
</dependency>
```

监听方式：

```java
@KafkaListener(
        topics = "${tmc.server.kafka.topic}",
        groupId = "${tmc.server.kafka.group-id}",
        concurrency = "${tmc.server.kafka.concurrency}"
)
public void onMessage(String message) {
    accessEventConsumer.consume(message);
}
```

说明：

- listener 层保持薄，只负责把 message 交给 `AccessEventConsumer`。
- 业务逻辑写在普通类中，便于单元测试。
- 初期使用手动 JSON 解析，保持和 SDK 上报的 JSON line 协议一致。

## 9. 指标设计

建议在 `TmcServerMetrics` 中维护以下指标：

| 指标 | 含义 |
|---|---|
| `messagesConsumed` | Kafka 消费到的消息数 |
| `messagesInvalid` | JSON 或字段非法消息数 |
| `messagesFailed` | 消费处理异常数 |
| `accessEventsAccumulated` | 成功进入热度累加器的事件数 |
| `mappingRuns` | 映射任务执行次数 |
| `mappingFailed` | 映射任务失败次数 |
| `trackedApps` | 当前追踪的 app 数量 |
| `trackedKeys` | 当前追踪的 key 数量 |
| `hotKeysDetected` | 最近一次探测出的热点 key 数量 |

这些指标用于排查：

- Kafka 有消息但 `accessEventsAccumulated` 不增长：可能 JSON 协议不匹配。
- `messagesInvalid` 很高：SDK 和服务端模型可能不一致。
- `trackedKeys` 持续上涨：需要增加过期清理策略。
- `hotKeysDetected` 长期为 0：阈值可能过高，或访问事件没有进入服务端。

## 10. 过期与内存控制设计

服务端热点探测会为每个出现过的 key 保存时间轮。

为了避免历史 key 无限增长，需要在映射任务中清理冷 key：

```text
wheel.sum() == 0
  -> 当前 key 最近一个完整窗口没有访问
  -> 从 app 的 wheelMap 中删除
```

清理时机：

```text
每次 app 映射任务完成后
  -> 遍历 wheelMap
  -> 删除 sum == 0 的 key
```

这样 key 在最近 30 秒完全没有访问后，会被自然清理。

## 11. 并发设计

### 11.1 消费线程与映射线程隔离

Kafka listener 线程负责消费事件。

映射任务由 Spring scheduler 或独立 `ScheduledExecutorService` 触发。

两者通过 `AccessEventAccumulator` 交互：

```text
Kafka listener thread
  -> accumulator.add(event)

Mapping scheduler thread
  -> accumulator.drainApp(appName)
```

### 11.2 降低锁竞争

`drainApp(appName)` 推荐使用原子替换：

```text
oldMap = appCounters.put(appName, new ConcurrentHashMap<>())
```

这样映射线程处理 `oldMap` 时，消费线程已经可以写入新的 map。

### 11.3 App 级串行

同一个 app 的映射任务建议串行执行。

原因：

- 同一个 app 的时间轮表是共享状态。
- 串行更容易保证时间片推进正确。

不同 app 后续可以并行映射，但本阶段可以先用单线程调度实现正确性。

## 12. 测试设计

### 12.1 `AccessEventConsumerTest`

覆盖：

3. 缺少 `appName` 的消息会被忽略。
4. 缺少 `key` 的消息会被忽略。
5. `weight <= 0` 的消息会被忽略。
6. `operation != GET` 的消息会被忽略。

### 12.2 `AccessEventAccumulatorTest`

覆盖：

1. 同一个 app/key 多次访问会累加。
2. 不同 app 的同名 key 互不影响。
3. `drainApp(appName)` 返回当前累计值。
4. `drainApp(appName)` 后再次 drain 返回空。
5. drain 后新事件进入新周期。

### 12.3 `TimeWheelCounterTest`

覆盖：

1. `addBucket(1)` 后 `sum()` 为 1。
2. 写入 10 个 bucket 后 `sum()` 为 10 个值的总和。
3. 第 11 次写入会覆盖第 1 个 bucket。
4. 写入 0 会推动窗口滑动并清除旧热度。

### 12.4 `HotKeyDetectorTest`

覆盖：

1. 低于阈值的 key 不会进入快照。
2. 达到阈值的 key 会进入快照。
3. 多个 key 按 score 倒序排列。
4. 结果最多返回 TopN。
5. `HotKey` 中 `appName`、`key`、`score`、`detectedAt`、`ttlMillis` 正确。

### 12.5 `HotKeyMappingSchedulerTest`

覆盖：

1. 当前周期访问量会写入时间轮。
2. 当前周期没有访问的旧 key 会写入 0。
3. 连续映射可以形成 30 秒滑窗。
4. `sum == 0` 的 key 会被清理。
5. 每次映射后 repository 保存最新 `HotKeySnapshot`。

3. 向应用服务器 rsyslog 写入多条同一 key 的访问事件。
4. 等待一个映射周期。
5. 查询 `HotKeySnapshotRepository` 或调试日志。
6. 确认目标 key 出现在热点快照中。

示例事件：

```json
{"appName":"product-service","key":"product:10001","timestamp":1720000000000,"weight":1,"clientId":"smoke-1","operation":"GET"}
```

当同一 `appName + key` 在 30 秒窗口内累计热度达到阈值后，服务端应生成类似快照：

```json
{
  "version": "product-service-1720000003000",
  "appName": "product-service",
  "hotKeys": [
    {
      "appName": "product-service",
      "key": "product:10001",
      "score": 120,
      "detectedAt": 1720000003000,
      "ttlMillis": 300000
    }
  ]
}
```
